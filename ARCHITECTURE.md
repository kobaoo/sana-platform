# Архитектура бэкенда

Этот репозиторий является примером архитектуры бэкенда для образовательной платформы с ролевым доступом, построенной на Encore.go. Содержит полноценную реализацию сервиса `organizations` и служит образцом для остальных сервисов.

## Структура проекта

```
/orgstructure              — группа сервисов (просто директория)
  /organizations           — сервис организаций
  /users                   — (будущий) сервис пользователей
  /employees               — (будущий) сервис сотрудников
/db                        — сервис базы данных (единая БД)
  db.go                    — объявление sqldb.NewDatabase
  atlas.hcl                — Atlas-конфиг, собирает ent-схемы всех сервисов
  /migrations              — все SQL-миграции
```

## Основные принципы

### Единая база данных
- Все сервисы работают с одной БД, объявленной в `db/db.go`.
- Каждый сервис ссылается на БД через `var db = sqldb.Named("lms")` — Encore-стандарт для межсервисного доступа.
- Миграции лежат только в `db/migrations/`.

### Сервисы-группы
- Верхнеуровневые директории (`orgstructure/`, ...) — группы сервисов, не Go-пакеты.
- Каждая поддиректория (`orgstructure/organizations/`) — отдельный Encore-сервис.
- Сервис отвечает за один бизнес-домен.
- Не размещать логику одного домена в другом сервисе.

### Ent-схемы
- Каждый сервис описывает свои таблицы в `<сервис>/ent/schema/`.
- Файлы внутри `ent/` кроме `ent/schema/` — генерируются автоматически, не трогать руками.
- Atlas в `db/atlas.hcl` собирает схемы со всех сервисов и генерирует единую миграцию.

## Слои сервиса

### Типы (`<service>/<service>_types.go`)
- Доменные модели, DTO запросов/ответов и типизированные константы.
- Не размещать здесь SQL-запросы, HTTP-логику или бизнес-правила.

### Эндпоинты (`<service>/<service>.go`)
- Содержит ссылку на БД, все API-эндпоинты и приватные хелперы.
- Файл делится ровно на три секции через баннер-комментарии:
  ```
  // ════ DATABASE ════   — var db = sqldb.Named("lms")
  // ════ ENDPOINTS ════  — encore:api функции
  // ════ INTERNAL ════   — приватные DB-хелперы
  ```
- Каждый эндпоинт строго следует порядку: проверка роли → валидация → вызов DB-хелпера → возврат ответа.
- Не размещать SQL-запросы напрямую в теле эндпоинта — только через хелперы секции INTERNAL.

### Тесты (`<service>/<service>_test.go`)
- Интеграционные тесты, работающие с реальной БД через тестовую инфраструктуру Encore.
- Не использовать моки базы данных.

## Правила роутинга

- Каждый сервис регистрирует свои маршруты через аннотации `//encore:api`.
- Ролевые зоны явные: `admin`, `hr`, `employee`.
- Проверка роли — всегда первое, что делает эндпоинт.
- Публичные эндпоинты помечаются `//encore:api public`, авторизованные — `//encore:api auth`.

## Правила данных

- SQL-параметры только через плейсхолдеры `$1, $2` — никакой конкатенации строк.
- Никогда не использовать `SELECT *` — только явный список колонок в том же порядке, что и `Scan`.
- Частичные обновления реализуются через `COALESCE($n, column_name)`.
- Удаление всегда мягкое: `is_active = FALSE`, без физического `DELETE`.

## Правила состояния и ошибок

- Все ошибки оборачиваются через `errs.B().Code(...).Msg(...).Err()`.
- Непредвиденные ошибки БД включают `.Cause(err)` для отладки.
- Пустые коллекции возвращаются как `[]T{}`, а не `nil`.

## Путь расширения

При добавлении нового сервиса:

1. Создать директорию `<группа>/<сервис>/`.
2. Описать ent-схему в `<сервис>/ent/schema/`.
3. Добавить путь к схеме в `db/atlas.hcl`.
4. Сгенерировать миграцию: `cd db && atlas migrate diff --env local`.
5. Определить типы в `<сервис>/<сервис>_types.go`.
6. Реализовать эндпоинты и DB-хелперы в `<сервис>/<сервис>.go`.
7. Покрыть каждый эндпоинт интеграционными тестами в `<сервис>/<сервис>_test.go`.
8. Следовать порядку секций `DATABASE → ENDPOINTS → INTERNAL`.
